Purpose & Scope

Here is described the goal of Emergency Quest project.

Emergency Quest is a system that integrates sensors with a powerful server application, making all processed informations available to a mobile app in which users can log-in and see their data.

The goal of the project is to improve dementia/Alzheimer affected people's life, making them able to continue living at home by themselves and helping them manage some of the potential risks around the home. They will wear a bracelet with "agitation" sensors and a GPS tracker, which data will be processed by the server and will be available to patient's caregivers by logging-in from the mobile application.

All around the house (in dangerous positions, i.e. doors or windows) motion sensors will be placed, so when the patient gets close to something dangerous, the system can play pre-recorded messages to avoid unpleasant situations. And if the patient succeeds to run away from the house, GPS tracker will notice it and send a notification in the mobile app to the caregivers.

Furthermore, "agitation" is a common state for people affected by dementia/Alzheimer, so the system can recognize this particular state and change the environment to relax the patient, by controlling lights and playing music. If these actions don't succeed, the system will understand that is a case of extreme conditions, so will automatically lock doors and windows, sending a notification to the mobile app, trying to avoid accidents.

AmI

Here are listed the principal AmI features.

AmI Main Steps

AmI Step Description
Sensing Motion and agitation will be detected. We will also use a GPS locator.
Reasoning Ability of switching decisions between cases of motion problems, agitation problems (even detecting extreme conditions) or escaping from house problems.
Acting The system will try to guide people affected by dementia by playing useful messages. Otherwise, in case of agitation, the system will try to change environment to make it more comfortable (lights and music). In case of extreme conditions, it will proceed locking doors and windows and sending notifications to caregivers.
Interacting The system will communicate with patient, family and environment. An app accessible by both users will be available for relatives to track patient movements and to access previously stored sensor data. Furthermore, the mobile app will receive all types of notifications.

AmI Features

AmI Feature Description
Sensitive The system will process motion and agitation sensor data, and it will also process GPS tracking data.
Responsive The system will respond to elderly user affected by dementia/Alzheimer needs, adapting also the environment in case of agitation mild or extreme symptoms.
Adaptive The system is able to adapt to the context (motion, agitation or escaping problems), detecting it by what it senses.
Transparent The end user will not need any further knowledge to use the system.
Ubiquitous Both patients and family components will always have access to motion data communicating with a server, within an application.
Intelligent The system can estimate the difference between a critical situation and a situation of mild symptoms.

Informations

Here you can find more info about words used in documentation and actors involved.

Glossary

  • Bracelet: system to monitor patient health state and his GPS position, its function is to improve interaction between environment and end user.

  • Default: default settings when the system is configured for the first time [SETUP].

  • Doctor: patient's physician, whose work can be simplified from the interaction with the system.

  • Extreme condition: condition in which base actions of the system can't satisfy patient needs, it is a dangerous condition.

  • Map: geographic map on which there is current position of the patient, with different colored placeholders to indicate previous positions of the patient himself.

  • Notification: sent by the server to the users, with two different types of it: normal notices and warning notices.

  • Perimeter: area set from patient caregivers, from which patient can't go out.

  • Preferences: personalized settings for patient caregivers or patient himself, to improve default settings, like the registration of messages with their own voice, setting lights and music.

  • Room: environment where the system is installed and where this one interacts with the user, i.e. where sensor data are collected.

  • Setup: set of parameters chosen by caregivers or patients which personalizes the behavior of the app.

  • System: all parts that shape Emergency Quest, divided in:
    1. APP: tool for interaction between system and users;
    2. SERVER: sensor data storage and processing;
    3. ROOM: sensing and acting.

Actors

  • User affected by dementia
  • User affected by Alzheimer
  • Caregiver of a patient affected by above deseases
  • Dementia/Alzheimer user's doctor

System Requirements

Here are listed functional and non-functional requirements of the system.

Functional Requirements

Functional Area Description
SETUP First usage of the system and its configuration.
APP How the application is made.
SERVER How the remote server is configured and how it works.
ROOM How the room senses and interacts with the user.

  • PRIORITY 1
    • SETUP.1: subscription from the application, creating a new user in the database by registration of the code given with the bracelet.
    • SETUP.2: no-trespassing perimeter setting.

  • PRIORITY 2
    • SETUP.3: registration of messages to play when the patient gets close to a danger situation.

  • PRIORITY 3
    • SETUP.4: default configuration of lights color and type of music and songs to relax patient.
    • SETUP.5: default configuration of pre-recorded messages to play to the patient.

  • PRIORITY 4
    • SETUP.6: possibility to insert doctor's e-mail to send him informations to access the account and monitor the patient.

  • PRIORITY 1
    • APP.1: log-in required for using every single feature of the system.
    • APP.2: presence of a map in which the position of the patient is shown with a placeholder.
    • APP.3: notifications sent in the app in a dangerous condition for the patient.

  • PRIORITY 2
    • APP.4: setting of particular messages that will be played at predefined times to remind the patient to do something (like taking the medicines, etc..).
    • APP.5: possibility for the caregivers to unlock doors from the app in case of extreme condition of the patient.
    • APP.6: receive a notification if the patient takes off the bracelet.
    • APP.7: inserting on the map previously recorded (during the day) positions of the patient with a different colored placeholder.

  • PRIORITY 3
    • APP.8: grant users the access to all data previous stored.
    • APP.9: users can delete data previous stored if they don't need it anymore.
    • APP.10: possibility to set preferences for light colors and type of music (in relaxing case).

  • PRIORITY 4
    • APP.11: grant to the doctor access to all data stored, to better monitor his patient.

  • PRIORITY 5
    • APP.12: give the user the possibility to automatically delete previous stored data after a predefined time.

  • PRIORITY 1
    • SERVER.1: presence of a database in which all user data is stored (log-in informations and sensor data related to the patient).
    • SERVER.2: always running application that processes data received from sensors and stores them in the correct position (in the next requirements we always refer to the application).
    • SERVER.3: processes GPS data and gives the user the possibility to always access patient current position.
    • SERVER.4: selects the right action to do, distinguishing when it's necessary to relax the patient or to lock doors and windows in a dangerous condition (processing sensor data).
    • SERVER.5: when it starts relaxing the patient, it plays the correct instructions using microphones in the room.

  • PRIORITY 2
    • SERVER.6: traces previous GPS positions of the patient and makes them accessible from the mobile app.
    • SERVER.7: locks doors and windows in case of extreme patient's conditions, if base actions can't help him.

  • PRIORITY 3
    • SERVER.8: stores user's preferences in the database.
    • SERVER.9: detects if the patient takes off the bracelet and sends a notification to the mobile app.

  • PRIORITY 1
    • ROOM.1: usage of a bracelet with sensors to monitor patient's agitation and with GPS to monitor his position.
    • ROOM.2: presence in the house of sensors to monitor patient's motion (and letting the server application act in case of danger).

  • PRIORITY 2
    • ROOM.3: bracelet and sensors are connected with the Wi-Fi to easily communicate with the server.

  • PRIORITY 3
    • ROOM.4: some information can be partially processed directly from sensors, without interaction with server.

  • PRIORITY 4
    • ROOM.5: bracelet connects automatically to house Wi-Fi connection by setting the password in the mobile app.


Non-Functional Requirements

    • PORTABILITY
  • NFR 2: in the user home, some sort of controllable lights must be available.
    • INTEROPERABILITY
  • NFR 3: localization language in English and Italian (and next Spanish).
    • USABILITY
  • NFR 4: always running server, able to notify 24/7 the user in case of dangerous conditions.
    • SYSTEM READINESS (PERFORMANCE)
